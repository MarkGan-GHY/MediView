# 服药提醒推送功能 — 开发记录

## 一、修改目的

在本次修改前，手机端的「服药提醒」模块只完成了前端 UI 与本地持久化：用户可以添加、编辑、删除提醒，数据通过 SharedPreferences 保存。**但提醒到达指定时间后，系统不会做任何事**——既没有本地通知，也没有任何向眼镜端推送的链路。

本次修改要补齐三块核心能力：

1. **定时检测**：让手机在用户设定的服药时间点真正"知道时间到了"
2. **跨端推送**：把"到时间了"这件事传递给眼镜端
3. **眼镜显示**：在眼镜上展示一个清晰的提醒画面，并支持用户确认

最终目标：用户在手机上设好的提醒，到点会自动在眼镜上弹出，用户做一个手势即可确认服药。

---

## 二、修改方式

### 2.1 架构约束与选择

本项目原有架构约定：**手机做 HTTP Server，眼镜做 HTTP Client**。眼镜通过 NSD（mDNS）+ UDP 广播发现手机服务，主动向手机发请求。这种方向不便由手机主动 push 推送给眼镜。

因此本次采用**「手机维护待推送队列 + 眼镜定期轮询」**方案：

- 手机端在闹钟触发时，把一条「待推送提醒消息」放进队列
- 眼镜端每 30 秒查询一次手机的队列
- 一旦发现新消息，眼镜端立刻拉起一个全屏提醒页
- 用户在眼镜端做手势确认后，眼镜回调手机将该消息从队列移除

### 2.2 关键技术选型

| 问题 | 选用方案 | 备选/原因 |
|------|----------|-----------|
| 定时触发 | `AlarmManager.setExactAndAllowWhileIdle` | 比 `WorkManager` 更精确，能在 Doze 模式下唤醒 |
| 推送方向 | 眼镜轮询（pull） | 受限于现有"眼镜=Client"架构；轮询足以满足提醒级实时性 |
| 队列持久化 | 内存 + SharedPreferences | 保证 App 进程被杀后队列可恢复 |
| 跨端传输 | 复用现有 NanoHTTPD + JSON | 与 `/analyzeDrug` 一致风格，扩展成本低 |
| 眼镜 UI | 单独的 Activity（继承 `BaseMirrorActivity`） | 与 `DrugCaptureActivity` 解耦，避免打断拍照流程 |

### 2.3 实现顺序

按依赖关系从底层往上：

1. 手机端：`ReminderMessage` DTO + `ReminderPushQueue` 队列
2. 手机端：`ReminderScheduler` + `ReminderAlarmReceiver`（接闹钟）
3. 手机端：在 `LocalHttpServer` 增加两个 HTTP 接口
4. 手机端：让 `ReminderViewModel` 在 CRUD 时调用 Scheduler
5. 眼镜端：`ReminderApiClient`（HTTP 客户端） + `ReminderActivity` + 布局
6. 眼镜端：`ReminderPollingService`（前台轮询服务），从 `MyApplication` 启动

## 三、修改内容清单

### 3.1 手机端（phone_app）

#### 新增文件

| 文件 | 说明 |
|------|------|
| `model/ReminderMessage.kt` | 推送消息数据类，字段：`messageId / reminderId / name / timesPerDay / amount / unit / scheduledTime / enqueuedAt` |
| `reminder/ReminderPushQueue.kt` | 单例队列，进程内同步 + SharedPreferences 持久化，提供 `enqueue / snapshot / ack`；超过 30 分钟未取走的消息自动过期 |
| `reminder/ReminderScheduler.kt` | 封装 AlarmManager；按 `MedicineReminder.times × repeatDays` 注册每个时间点的精确闹钟；提供 `reschedule / cancel / rescheduleAll / rescheduleSingle / computeNextTrigger` |
| `reminder/ReminderAlarmReceiver.kt` | 闹钟触发后：①构造 `ReminderMessage` 入队 ②立即重排该时间点的下次触发（因为一次性 alarm 不会自动循环） |

#### 修改文件

| 文件 | 改动 |
|------|------|
| `server/LocalHttpServer.kt` | 新增两个路由：`GET /pendingReminders` 返回队列快照；`POST /ackReminder?messageId=xxx` 移除已确认消息 |
| `viewmodel/ReminderViewModel.kt` | `init` 时全量重排；`add/update/delete/updateStatus` 每次都同步调用 Scheduler |
| `AndroidManifest.xml` | 新增权限 `SCHEDULE_EXACT_ALARM`、`USE_EXACT_ALARM`；注册 `ReminderAlarmReceiver` |

### 3.2 眼镜端（glass-app）

#### 新增文件

| 文件 | 说明 |
|------|------|
| `network/ReminderApiClient.kt` | OkHttp 封装；`fetchPending(host, port)` 拉取列表，`ack(host, port, messageId)` 确认 |
| `service/ReminderPollingService.kt` | 前台服务；持有 `PhoneServiceDiscovery` 获取手机地址；每 30 秒拉取一次；命中未弹过的消息则 `startActivity` 启动提醒页 |
| `ui/activity/reminder/ReminderActivity.kt` | 继承 `BaseMirrorActivity<ActivityReminderBinding>`，双目同步显示；单击调用 `ack` 后 `finish`，双击直接 `finish`（不 ack） |
| `res/layout/activity_reminder.xml` | 全屏黑底卡片：标题、时间（大号）、药名、剂量、底部提示语 |

#### 修改文件

| 文件 | 改动 |
|------|------|
| `MyApplication.kt` | 在 `onCreate` 中通过 `ReminderPollingService.start(this)` 启动轮询服务 |
| `AndroidManifest.xml` | 新增权限 `FOREGROUND_SERVICE`、`FOREGROUND_SERVICE_REMOTE_MESSAGING`、`POST_NOTIFICATIONS`；注册 `ReminderActivity` 与 `ReminderPollingService` |

## 四、最终逻辑

### 4.1 数据流总览

```
┌─────────────────── 手机端 ────────────────────┐         ┌─────────── 眼镜端 ───────────┐
│                                                │         │                                │
│  用户在 ReminderScreen 添加提醒                │         │                                │
│         ↓                                      │         │                                │
│  ReminderViewModel.addReminder                 │         │                                │
│         ↓                                      │         │                                │
│  ReminderScheduler.reschedule                  │         │                                │
│         ↓                                      │         │                                │
│  AlarmManager.setExactAndAllowWhileIdle        │         │                                │
│         ↓ （到达提醒时间）                     │         │                                │
│  ReminderAlarmReceiver.onReceive               │         │                                │
│         ├──→ ReminderPushQueue.enqueue         │         │                                │
│         └──→ ReminderScheduler.rescheduleSingle│         │                                │
│              （注册下一次同时间点）            │         │                                │
│                                                │         │                                │
│  LocalHttpServer                               │ ←HTTP→  │  ReminderPollingService        │
│   GET /pendingReminders ←──────────────────────┼─────────┤   每 30s 轮询                  │
│   返回队列 JSON                                │         │           ↓                    │
│                                                │         │   命中新消息 → startActivity   │
│                                                │         │           ↓                    │
│                                                │         │   ReminderActivity 双目显示    │
│                                                │         │           ↓                    │
│   POST /ackReminder?messageId=xxx ←────────────┼─────────┤   用户单击 → 调用 ack          │
│   队列移除                                     │         │           ↓                    │
│                                                │         │   finish() 返回 DrugCapture    │
└────────────────────────────────────────────────┘         └────────────────────────────────┘
```

### 4.2 手机端时序

1. **CRUD 时同步重排**
   用户每次添加、修改、删除提醒，或切换启用开关，`ReminderViewModel` 都会调用 `ReminderScheduler.reschedule()`：先按 `requestCode = reminderId.hashCode() * 100 + timeIndex` 取消该提醒的所有旧 alarm，再按当前 `times × repeatDays` 重新注册每个时间点的下次触发。

2. **闹钟触发**
   `AlarmManager` 到点唤醒 `ReminderAlarmReceiver`，Receiver 在 `onReceive` 中：
   - 构造 `ReminderMessage`（`messageId` 用 UUID），调用 `ReminderPushQueue.enqueue` 入队；队列同时写入 SharedPreferences
   - 调用 `ReminderScheduler.rescheduleSingle` 计算该时间点的下次触发（根据 `repeatDays` 找下一个匹配的日期），重新注册 alarm

3. **HTTP 服务**
   - `GET /pendingReminders` 调用 `ReminderPushQueue.snapshot(context)`，顺手清理超过 30 分钟未被取走的过期消息，返回 `{"success":true,"reminders":[...]}`
   - `POST /ackReminder?messageId=xxx` 调用 `ReminderPushQueue.ack(context, messageId)` 从队列移除该条

### 4.3 眼镜端时序

1. **服务启动**
   `MyApplication.onCreate` → `ReminderPollingService.start(this)` 启动前台服务（独立于 `DrugCaptureActivity` 生命周期，即使用户在拍照页或退出拍照页都不影响轮询）。

2. **服务发现**
   轮询服务持有一个 `PhoneServiceDiscovery` 实例，通过 NSD + UDP 双通道发现手机 `host:port`。未发现时该轮跳过 HTTP 请求。

3. **轮询循环**
   每 30 秒调用 `ReminderApiClient.fetchPending(host, port)`：
   - 拿到响应后，取队列中"本地 `shownMessageIds` 集合里没有"的第一条
   - 加入 `shownMessageIds`（防止确认前重复拉起 Activity）
   - 用 `Intent + FLAG_ACTIVITY_NEW_TASK` 启动 `ReminderActivity`，把消息字段和手机地址打包传过去

4. **提醒展示与交互**
   `ReminderActivity` 在 `onCreate` 中通过 `mBindingPair.updateView` 双目同步显示药名、时间、剂量；手势监听复用 `templeActionViewModel.state.collect` 的模式：
   - **单击** → 调用 `ReminderApiClient.ack` 后 `finish()`
   - **双击** → 不 ack 直接 `finish()`；由于 `shownMessageIds` 还在内存里，30s 后下一轮不会立即重弹；但若 30 分钟内队列未过期，App 重启 / shownMessageIds 失效后仍会再次弹出

5. **重复保护**
   - 手机侧：同一 `reminderId + scheduledTime` 已在队列时跳过重复入队
   - 眼镜侧：本地 `shownMessageIds` 集合避免同一 messageId 在确认前被多次拉起；集合容量超过 200 时压缩到最近 100 条

### 4.4 失败与恢复

| 场景 | 行为 |
|------|------|
| 眼镜断连 / 手机服务未启动 | 轮询请求失败，本轮跳过；下一轮自动重试 |
| 手机进程被杀 | 队列已落盘 SharedPreferences；下次 `ReminderViewModel` 创建时 `init` 调用 `rescheduleAll` 恢复所有 alarm |
| 队列里消息长期没人取（眼镜长时间离线） | `snapshot` 清理超过 30 分钟的消息，避免无限堆积 |
| Android 12+ 用户拒绝精确闹钟权限 | `ReminderScheduler.setExact` 检测到 `canScheduleExactAlarms == false` 时，降级为 `setAndAllowWhileIdle`（精度降到 ~10 分钟） |

### 4.5 设计权衡

- **轮询而非主动推送**：保持现有架构不变，眼镜端继续作为 Client；代价是 0–30 秒的延迟
- **不弹本地通知**：手机端 alarm 触发后只入队、不弹手机通知；这是因为产品定位是"提醒在眼镜上展示"。如需手机也提示，可在 `ReminderAlarmReceiver` 中追加 `NotificationManager.notify`
- **未做开机自启**：暂未注册 `BOOT_COMPLETED` Receiver；用户重启手机后需要打开一次手机 App 才会重排 alarm。如需自启，新增 BootReceiver 调用 `ReminderScheduler.rescheduleAll` 即可

## 五、自检要点

- 加一条"1 分钟后"的提醒 → 等 1 分钟 → Logcat 应看到 `ReminderAlarmReceiver: 闹钟触发`
- 浏览器 / curl 访问 `http://<手机IP>:8080/pendingReminders` → 应返回包含该消息的 JSON
- 眼镜在 30 秒内弹出提醒页 → 单击 → 手机 Logcat 看到 `眼镜确认提醒`，队列清空
- 双击"稍后" → 提醒页关闭 → 但本地集合记录已展示，不会立即重弹
- 配置一条提醒含两个时间点（如 08:00 / 12:00）→ 两个时间点都应独立触发
