# 手机端 Prompt（给 CLI Agent）

你正在修改一个 Android Kotlin 项目。
目标不是重写整个通信系统，而是**将当前“手机端作为 HTTP 服务端、眼镜端通过硬编码 IP 访问”的模式，迁移为“手机端发布可被局域网发现的服务”**，为后续眼镜端接入服务发现做准备。

---

## 一、项目目标

当前系统存在以下问题：

* 手机热点每次重启后，手机的局域网 IP 可能变化
* 眼镜端如果硬编码手机 IP，会导致连接失效
* 当前 HTTP 通信虽然可用，但“寻址方式”不稳定

本次只改**手机端**，实现以下目标：

1. 保留现有手机端 HTTP 服务能力，尽量不破坏已有业务接口
2. 新增一个**服务发布层**
3. 手机端启动本地 HTTP 服务后，自动通过 **Android NSD（Network Service Discovery）** 将该服务发布到局域网
4. 服务发布信息至少包含：

   * serviceName
   * serviceType
   * port
5. 为后续眼镜端自动发现手机服务打基础
6. 本次**不要实现眼镜端发现逻辑**
7. 本次**不要切换为 WebSocket、MQTT 或云端**
8. 以“最小侵入改造”为原则，优先复用现有 HTTP Server

---

## 二、改造原则

请严格遵守以下原则：

### 1. 不重写现有业务协议

如果当前手机端已有 HTTP API：

* 保留原有 endpoint
* 保留原有请求/响应格式
* 保留原有处理逻辑

本次只解决：

* “服务如何被找到”
  而不是：
* “业务接口如何重构”

---

### 2. 将“通信服务”和“服务发现发布”解耦

请拆成两个相对独立的模块：

#### A. 本地 HTTP 服务模块

职责：

* 启动/停止 HTTP 服务
* 暴露当前实际监听端口
* 不关心 NSD

#### B. NSD 服务发布模块

职责：

* 使用 `NsdManager` 注册服务
* 使用 serviceType / serviceName 对外广播
* 处理注册成功、失败、注销等生命周期
* 不关心业务 API

---

### 3. 避免把逻辑写死在 Activity 里

不要把核心启动逻辑散落在页面中。
优先放在以下位置之一：

* `Application`
* 前台 `Service`
* 专门的 manager / repository / controller

如果当前项目已有统一的连接管理器或网络管理器，请在现有架构中扩展，而不是另起一套平行体系。

---

### 4. 默认使用 Kotlin

整个实现使用 Kotlin。
除非项目中已有必须兼容的 Java 文件，否则不要新增 Java 实现。

---

## 三、实现要求

请完成以下改造。

---

### 1. 为手机端新增 NSD 服务发布能力

使用 Android 原生 `NsdManager` 实现服务注册。

建议参数：

* `serviceType = "_rayneo-pill._tcp."`
* `serviceName = "rayneo_phone_bridge"`

但请将它们提取为可配置常量，不要硬编码在业务逻辑中。

要求：

* 支持注册服务
* 支持注销服务
* 支持监听注册成功 / 失败
* 如果系统自动改写 serviceName，要正确记录最终 serviceName
* 输出清晰日志

---

### 2. 保留并封装当前 HTTP Server

如果项目中已经有 HTTP Server：

* 不要删除
* 不要大改 endpoint
* 将其整理为一个清晰的 `LocalHttpServerManager` 或类似类

要求：

* 支持 `start()`
* 支持 `stop()`
* 支持获取实际监听端口
* 如果端口冲突，能够报错或自动回退到可用端口（若当前项目已有固定端口要求，则至少做出清晰报错）

---

### 3. 启动顺序必须明确

推荐顺序：

1. 启动本地 HTTP Server
2. 获取实际端口
3. 通过 NSD 发布服务
4. 在应用退出或服务关闭时注销 NSD 服务

不要先发布服务再启动 HTTP Server。

---

### 4. 增加统一生命周期控制器

请新增一个统一入口，例如：

* `PhoneBridgeServiceController`
* `PhoneServerCoordinator`
* 或与现有架构一致的命名

职责：

* 启动 HTTP Server
* 启动 NSD 注册
* 管理关闭顺序
* 提供当前运行状态：

  * serverRunning
  * published
  * port
  * serviceName
  * serviceType

---

### 5. 增加状态与日志输出

至少输出以下日志：

* HTTP 服务启动成功，端口是多少
* HTTP 服务启动失败，原因是什么
* NSD 注册开始
* NSD 注册成功（最终 serviceName / port / serviceType）
* NSD 注册失败（errorCode）
* NSD 注销成功 / 失败

日志要统一 tag，便于 adb 过滤。

---

### 6. 增加本地可观察状态

为了便于 UI 或调试页面后续接入，请暴露一个状态流，优先使用：

* `StateFlow`
  或
* `MutableStateFlow`

建议状态字段：

```kotlin
data class PhoneBridgePublishState(
    val serverRunning: Boolean = false,
    val published: Boolean = false,
    val port: Int? = null,
    val serviceName: String = "",
    val serviceType: String = "",
    val errorMessage: String? = null
)
```

要求：

* HTTP 启动后更新状态
* NSD 注册成功后更新状态
* 失败时更新错误信息

---

如果项目已有更合适的分层，请融入原有结构，不要机械照搬。

## 五、实现细节要求

### 1. NsdServicePublisher

请实现一个独立类，负责：

* 持有 `NsdManager`
* 构建 `NsdServiceInfo`
* 注册服务
* 注销服务
* 管理 `RegistrationListener`

建议接口：

```kotlin
interface ServicePublisher {
    fun register(port: Int)
    fun unregister()
}
```

或直接类实现：

```kotlin
class NsdServicePublisher(...)
```

---

### 2. LocalHttpServerManager

请将现有 HTTP 服务整理成可复用管理器，建议接口：

```kotlin
interface LocalServerManager {
    fun start(): Result<Int>   // 返回实际端口
    fun stop()
    val isRunning: Boolean
}
```

如果项目当前 server 是阻塞式启动，请安全地改造成后台线程或协程管理，避免阻塞主线程。

---

### 3. PhoneBridgeServiceController

建议提供：

```kotlin
fun start()
fun stop()
val state: StateFlow<PhoneBridgePublishState>
```

行为要求：

* `start()` 时：

  * 启动 HTTP server
  * 成功后注册 NSD
* `stop()` 时：

  * 先注销 NSD
  * 再关闭 HTTP server

---

### 4. 线程与协程

要求：

* 不要在主线程直接做阻塞式网络监听
* 如果当前 HTTP server 是阻塞启动，请放到 `Dispatchers.IO`
* 状态更新回到安全线程
* 代码要避免明显的生命周期泄漏

---

### 5. Manifest 与权限检查

请检查并补充手机端所需内容，但不要无脑乱加权限。

需要你：

* 检查当前项目是否已经具备网络相关权限
* 如确实需要，请补充必要权限
* 如 Android 版本差异会影响 NSD 行为，请在代码注释中说明

---

## 六、不要做的事情

本次不要做以下改动：

* 不要修改眼镜端代码
* 不要实现眼镜端 discover / resolve
* 不要把通信协议整体从 HTTP 改成 WebSocket
* 不要引入重型第三方服务发现库，优先 Android 原生 NSD
* 不要为了“以后可能用到”而提前加入过多抽象层
* 不要把 demo 逻辑写进 Activity 再算完成
* 不要删除现有接口，除非已有明显错误且你能给出兼容替代

2. 现有业务层调用尽量少改
